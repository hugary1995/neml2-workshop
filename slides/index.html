Example 2: Broadcasting

  • Borrowing Numpy's definition of broadcasting:
    The term broadcasting describes how NumPy treats arrays with different shapes during arithmetic operations. Subject to certain constraints, the smaller array is “broadcast” across the larger array so that they have compatible shapes.
  • NEML2's broadcasting semantics is slightly different from what's used in Numpy and PyTorch: Two tensors are said to be broadcastable if
  1. They have the same base shape, AND
  2. Iterating backward from the last batch dimension, one the following is satisfied
    1. Both tensors have the same size on that dimension;
    2. One tensor has size 1 on that dimension;
    3. That dimension does not exist in one tensor.
  • The rules can be generalized to operations involving more than two tensors.

Example 2: Broadcasting

Examples 3 & 4: Tensor Indexing

If you are familiar with Python array indexing:

NEML2 tensor indexing works in pretty much the same way,
except that now indexing distinguishes between batch and base dimensions.

Example 3: Single-axis indexing

  • Integer: A[1], B[-2]
  • Boolean: A[[True, True, False]]
  • Slice: A[i:j:k]

Example 4: Multi-axis indexing

  • Mixed: A[1, 2:-1:3, [True, False, True]]
  • Ellipsis: A[2:5, ..., -3]
  • None: A[None], A[..., None]

Example 5: Tensor View

  • Memory efficiency is one of the most important factors that determine the optimal batch size (OBS).
  • Tensor views are created wherever possible.
  • View avoids explicit data copy, thus allows us to do fast and memory efficient reshaping, slicing and element-wise operations.

Example 7: Labeled Axis

The primary difficulty in working with tensor slices is due to their lack of physical meaning.

NEML2 provides a mechanism to assign human-readable labels to tensor indices.
The canonical object for working with labels is LabeledAxis.

Terminlogies

  • Item: A contiguous slice with an assigned label.
  • Variable: An item of primitive tensor type, e.g., Scalar, Vec, R2, etc.
  • Sub-axis: An item of type LabeledAxis

Example 8: Labeled Tensors

LabeledTensor is the base class for all labeled tensors.

  • The base class is templated on the number of tensor axes N.
  • A labeled tensor consists of one BatchTensor and N LabeledAxiss.
  • 3 specializations: LabeledVector, LabeledMatrix, LabeledTensor3D

LabeledTensors are primarily used to pass information between models (IOU the definition of model).

  • LabeledVector can be used to store a model's input and output.
  • LabeledMatrix can be used to store first derivatives.
  • LabeledTensor3D can be used to store second derivatives.

The Flying Ball

From example 9 to example 17, we will be throwing balls on the playground:

\[ \begin{aligned} \dfrac{\text{ d}x}{\text{ d}t} & = v \\ \dfrac{\text{ d}v}{\text{ d}t} & = g-\mu v, \quad g = [0, -9.81, 0]^T \end{aligned} \]

with initial conditions

\[ \begin{aligned} x(0) & = x_0 \\ v(0) & = v_0 \end{aligned} \]

Example 9: Model definition

In mathematics, a model can be written as

\[ \begin{align} \mathrm{y} = f(\mathrm{x}; \mathrm{p}) \end{align} \]

In the (time-discretized) flying ball example,

\[ \begin{align} \mathrm{x} & = \{x_n, v_n, t, t_n\} \\ \mathrm{y} & = \{x, v\} \\ \mathrm{p} & = \{\mu\} \end{align} \]

In NEML2, all models derive from Model.

Symbol NEML2 terminology Method
$\mathrm{x}$ input variable Model::declare_input_variable
$\mathrm{y}$ output variable Model::declare_output_variable
$\mathrm{p}$ parameter Model::declare_parameter
$f$ forward operator Model::set_value

In mathematics, a model can be written as

\[ \begin{align} \mathrm{y} = f(\mathrm{x}; \mathrm{p}) \end{align} \]

In the (time-discretized) flying ball example,

\[ \begin{align} \mathrm{x} & = \{x_n, v_n, t, t_n\} \\ \mathrm{y} & = \{x, v\} \\ \mathrm{p} & = \{\mu\} \end{align} \]

All models have an input axis and an output axis, both of type LabedAxis.

The following sub-axis naming conventions are used:

Sub-axis name Description
state Variables that describe the current (or trial) state of the system.
old_state Variables that describe the previous state of the system.
forces External variables that drives the motion/evolution of the system
old_forces External driving variables from the previous time step
residual Residual of the variables that define an implicit system.

Example 10: Registry, Factory, and Deserialization

NEML2 uses the standard registry-factory pattern for object self-registration and run-time manufacturing.

All manufacturable objects can be deserialized from an input file. NEML2 input files have extension .i and use the HIT hierarchical text language.

To create a custom manufacturiable object, simply

  • inherit from NEML2Object, and
  • define expected_options().

Input file top-level entry-points:

Section name Base class Description
[Models] Model Define input, output, parameters, and the forward operator of a model.
[Solvers] Solver Solve an (implicit) system of equations.
[Drivers] Driver Drive a (autonomous) system with given initial conditions and external forces.
[Tensors] torch::Tensor Wrappers for specifying tensors from the input file.
[Data] Data Store common data related to the system.

Example 12: Model Derivatives and Automatic Differentiation

  • Many "stiff" systems could benefit from implicit solution algorithms.
  • Providing accurate model derivatives (i.e. the Jacobian)
    • could accelerate the implicit solution process and
    • is mandatory to apply the implicit function theorem.
  • PyTorch/libtorch offers backward-mode (and experimental forward-mode) automatic differentiation (AD), which is useful in rapid prototyping. 
							
								[Models]
									[ball]
										type = FlyingBall
										velocity = 'state/v'
										acceleration = 'state/a'
										use_AD_first_derivative = true
										use_AD_second_derivative = true

Example 13: Model Composition, Dependency Resolution, and Chain Rule

So far we have implemented

\[ \begin{aligned} a(v) & = g - 0.01 v, \quad g = [0, -9.81, 0]^T \end{aligned} \]

We need 3 more equations to close the system:

\[ \begin{aligned} r_x(x, v, x_n, t, t_n) & = x - x_n - v \cdot (t - t_n) \\ r_v(v, v_n, t, t_n) & = v - v_n - a \cdot (t - t_n) \\ \{x, v\} & = \text{root}\left([r_x, r_v]^T\right) \end{aligned} \]

Relax -- these are all standard models that are offered by default.

  • The first 2 equations are VecBackwardEulerTimeIntegration.
  • The third equation is ImplicitUpdate.
  • But how do we "stitch" them together?
    • Automatic dependency registration
    • Automatic input/output identification
    • Automatic dependency resolution
    • Automatic chain rule

See the documentation page for more details.

Example 15: Driver

Well, we certainly don't want to write and compile some code every time we throw the ball with a different velocity.

That's the exact job of a Driver.
Here, we will implement our own BallLauncher by deriving from the TransientDriver.

The results can optionally be saved using zip-based pickling into a PyTorch archive.

  • The archive can be loaded inside Python using torch.jit.load.
  • All variables are saved as buffers (without backward propagation information).
  • The archive has two attributes: input and output. Each attribute is a dictionary with variable names as dictionary keys and variable values as dictionary values.

Example 16 & 17: Model Parametrization

Recall the problem definition

\[ \begin{align} \mathrm{y} = f(\mathrm{x}; \mathrm{p}) \end{align} \]

In the (time-discretized) flying ball example,

\[ \begin{align} \mathrm{x} & = \{x_n, v_n, t, t_n\} \\ \mathrm{y} & = \{x, v\} \\ \mathrm{p} & = \{\mu\} \end{align} \]

We have faithly implemented everything except $\mu$ -- it was hard-coded to be 0.01 of type Real.

That's not the ideal choice because

  • The code has to be recompiled every time the resistivity changes.
  • If the input and output can be batched, there is no reason why $\mu$ cannot be.
  • If the model is sent to a different device, the parameters should be sent to same device as well.
  • In many applications, the derivatives of the model response w.r.t. the parameters (or the action of the Jacobian on the parameters) are required.

Example 18: Setting up a Solid Mechanics Model

Solving the implicit system

$\{\sigma, \bar{\varepsilon}^p, K^p \} = \text{root}\left( r_\sigma, r_{\bar{\varepsilon}^p}, r_{K^p} \right)$
$r_\sigma = \sigma - \sigma_n - \dot{\sigma} \cdot (t - t_n)$
$r_{\bar{\varepsilon}^p} = \bar{\varepsilon}^p - \bar{\varepsilon}^p_n - \dot{\bar{\varepsilon}^p} \cdot (t - t_n)$
$r_{K^p} = K^p - K^p_n - \dot{K^p} \cdot (t - t_n)$

Elasticity

$\dot{\sigma} = \mathbb{C} : \dot{\varepsilon^e}$
$\dot{\varepsilon^e} = \dot{\varepsilon} - \dot{\varepsilon^p}$
$\dot{\varepsilon} = (\varepsilon - \varepsilon_n) / (t - t_n)$

Yield function

$\dot{\gamma} = \left( \dfrac{\lvert f^p \rvert}{\eta} \right)^n \text{heav}\left( f^p \right)$
$f^p = \bar{\sigma} - \sigma_y - h$
$\bar{\sigma} = \dfrac{3}{2} \lVert \text{dev}(M-X) \rVert$
$M = \sigma$
$X = H_k K^p$
$h = H_i \bar{\varepsilon}^p$

Plastic flow

$N_M = \partial f^p / \partial M$
$N_X = \partial f^p / \partial X$
$N_h = \partial f^p / \partial h$
$\dot{\varepsilon}^p = \dot{\gamma} N_M$
$\dot{\bar{\varepsilon}}^p = -\dot{\gamma} N_h$
$\dot{K}^p = -\dot{\gamma} N_X$
